Q41 Microservices Architecture - Building Distributed Systems Advanced
▼

🏗️ Service Decomposition

Breaking monoliths into independent services using domain-driven design. Each microservice owns its data, communicates via APIs, and can be deployed independently. Bounded contexts define service boundaries.

🚪 API Gateway Pattern

Single entry point for client requests that routes to appropriate microservices. Handles authentication, rate limiting, request transformation, and response aggregation. Simplifies client-service communication.

🕸️ Service Mesh

Infrastructure layer for service-to-service communication with features like load balancing, service discovery, circuit breaking, and observability. Tools: Istio, Linkerd, Consul Connect.

💬 Inter-Service Communication

Synchronous (REST, gRPC) and asynchronous (message queues, events) patterns. REST for simple queries, gRPC for performance, events for decoupling. Each pattern has trade-offs in consistency vs. availability.

🎯 Key Takeaways

  • Single Responsibility: Each microservice should do one thing well with its own database
  • API Gateway: Centralized entry point handles authentication, routing, and response aggregation
  • Service Independence: Services can be deployed, scaled, and updated independently
  • Failure Isolation: Use circuit breakers to prevent cascade failures across services
  • Data Consistency: Embrace eventual consistency with event-driven patterns (Saga pattern)
  • Service Discovery: Use tools like Consul, Eureka, or Kubernetes DNS for dynamic service location
  • Observability: Implement distributed tracing (Jaeger, Zipkin) and centralized logging (ELK)
  • Security: Use JWT tokens, mTLS for service-to-service communication, API Gateway for authentication

✅ DO's

  • Design services around business domains (DDD)
  • Use asynchronous messaging for loose coupling
  • Implement health checks and readiness probes
  • Version your APIs for backward compatibility
  • Use containerization (Docker) for consistency
  • Implement circuit breakers and retry logic
  • Monitor service dependencies and latencies
  • Use API Gateway for cross-cutting concerns

❌ DON'Ts

  • Don't share databases between services
  • Don't create too fine-grained microservices
  • Don't ignore network latency and failures
  • Don't use distributed transactions (use Sagas)
  • Don't forget about service versioning
  • Don't skip implementing correlation IDs
  • Don't ignore security between services
  • Don't over-engineer initially (start modular)
Q42 Microservices Service Discovery - Dynamic Service Location Advanced
▼

📋 Service Registry

Centralized database of service instances and locations. Services register on startup with metadata (host, port, health URL). Registry tracks available instances and removes unhealthy ones. Tools: Consul, Eureka, etcd, Zookeeper.

❤️ Health Checks

Regular heartbeat checks to verify service availability. Services expose /health endpoints returning status. Registry marks services as UP/DOWN based on check results. Failed checks trigger automatic deregistration.

⚖️ Client-Side Load Balancing

Client queries registry for service instances and chooses one using algorithms (round-robin, least connections). Eliminates single point of failure. Libraries: Ribbon (Netflix), Spring Cloud LoadBalancer.

🔄 Dynamic Discovery

Services automatically discover and adapt to topology changes. No hardcoded URLs. Enables auto-scaling and blue-green deployments. Client-side caching with TTL for performance. DNS-based (Kubernetes) or registry-based (Consul).

// npm install consul axios opossum
const express = require('express');
const Consul = require('consul');
const axios = require('axios');
const CircuitBreaker = require('opossum');

const consul = new Consul();

// Service Discovery Helper
class ServiceDiscovery {
    constructor() {
        this.cache = new Map();
        this.cacheTTL = 30000; // 30 seconds
    }

    async getServiceInstance(serviceName) {
        // Check cache first
        const cached = this.cache.get(serviceName);
        if (cached && Date.now() - cached.timestamp < this.cacheTTL) {
            return cached.instances[Math.floor(Math.random() * cached.instances.length)];
        }

        // Query Consul for healthy services
        const result = await consul.health.service({ service: serviceName, passing: true });
        
        if (result.length === 0) {
            throw new Error(`No healthy instances of ${serviceName} found`);
        }

        // Cache instances
        const instances = result.map(r => ({
            address: r.Service.Address,
            port: r.Service.Port,
            id: r.Service.ID
        }));
        
        this.cache.set(serviceName, { instances, timestamp: Date.now() });
        
        return instances[Math.floor(Math.random() * instances.length)];
    }

    async callService(serviceName, path, options = {}) {
        const instance = await this.getServiceInstance(serviceName);
        const url = `http://${instance.address}:${instance.port}${path}`;
        
        return axios({ url, ...options });
    }
}

const discovery = new ServiceDiscovery();

// Payment Service with Circuit Breaker
const app = express();
app.use(express.json());

// Circuit breaker for account service calls
const accountServiceCall = new CircuitBreaker(
    async (accountId) => {
        const response = await discovery.callService(
            'account-service',
            `/accounts/${accountId}`
        );
        return response.data;
    },
    {
        timeout: 5000,
        errorThresholdPercentage: 50,
        resetTimeout: 30000
    }
);

// Process payment with service discovery
app.post('/payments', async (req, res) => {
    try {
        const { fromAccount, toAccount, amount } = req.body;

        // Discover and call account service via circuit breaker
        const fromAccountData = await accountServiceCall.fire(fromAccount);
        
        if (fromAccountData.balance < amount) {
            return res.status(400).json({ error: 'Insufficient funds' });
        }

        // Call fraud-detection service
        const fraudCheck = await discovery.callService(
            'fraud-service',
            '/check',
            {
                method: 'POST',
                data: { fromAccount, toAccount, amount }
            }
        );

        if (fraudCheck.data.isFraud) {
            return res.status(403).json({ error: 'Transaction flagged as fraudulent' });
        }

        // Process payment (simplified)
        const paymentId = `PAY${Date.now()}`;
        
        res.status(201).json({
            paymentId,
            status: 'SUCCESS',
            fromAccount,
            toAccount,
            amount
        });
    } catch (error) {
        console.error('Payment failed:', error.message);
        res.status(500).json({ 
            error: 'Payment processing failed',
            details: error.message 
        });
    }
});

// Health check
app.get('/health', (req, res) => {
    res.json({ 
        status: 'UP', 
        service: 'payment-service',
        circuitBreaker: {
            accountService: accountServiceCall.opened ? 'OPEN' : 'CLOSED'
        }
    });
});

const PORT = 5001;
app.listen(PORT, async () => {
    console.log(`Payment Service on ${PORT}`);
    
    // Register with Consul
    await consul.agent.service.register({
        id: `payment-service-${PORT}`,
        name: 'payment-service',
        address: 'localhost',
        port: PORT,
        check: {
            http: `http://localhost:${PORT}/health`,
            interval: '10s'
        }
    });

🎯 Key Takeaways

  • Service Registry: Central database (Consul, Eureka) tracking all service instances with metadata
  • Health Checks: Regular heartbeats to detect unhealthy instances and remove them automatically
  • Client-Side Discovery: Clients query registry and choose instance (better performance, no SPOF)
  • Server-Side Discovery: Load balancer queries registry (simpler clients, centralized routing)
  • Caching: Cache service locations with TTL (30s-60s) to reduce registry load
  • Circuit Breaker: Prevent cascade failures when calling discovered services
  • DNS Integration: Kubernetes uses DNS for service discovery (service-name.namespace.svc.cluster.local)
  • Graceful Shutdown: Deregister from registry on shutdown to avoid routing to dead instances

✅ DO's

  • Implement health checks with appropriate intervals
  • Cache service locations with short TTL (30s)
  • Use unique service IDs (include PID or hostname)
  • Deregister services on graceful shutdown
  • Implement retry logic with exponential backoff
  • Tag services with metadata (version, region)
  • Monitor registry health and availability
  • Use circuit breakers for discovered services

❌ DON'Ts

  • Don't hardcode service URLs in code
  • Don't skip health check endpoints
  • Don't cache service locations indefinitely
  • Don't ignore deregistration on shutdown
  • Don't query registry for every request
  • Don't forget to handle no healthy instances
  • Don't ignore network partitions (split-brain)
  • Don't use service discovery for static deps
Q43 Event-Driven Architecture - Event Emitters in Node.js Intermediate
▼

📡 EventEmitter API

Core Node.js module for implementing observer pattern. Methods: on/addListener (register), emit (trigger), once (one-time), removeListener/off (unregister). Many Node.js APIs inherit from EventEmitter (Streams, HTTP, ChildProcess).

🎭 Custom Events

Create custom event emitters by extending EventEmitter class. Emit events with data payloads. Multiple listeners can subscribe to same event. Synchronous execution by default (listeners called in order).

⚠️ Error Handling

Always listen for 'error' events - unhandled errors crash process. Use try-catch in listeners. EventEmitter.captureRejections option for async errors. Process 'uncaughtException' as last resort.

🧠 Memory Leaks

Each EventEmitter has max listeners limit (default 10) to detect memory leaks. Warning emitted when exceeded. Use removeListener to clean up. Call setMaxListeners to increase limit if legitimate. Always remove listeners when done.

💻 Basic EventEmitter Usage & Custom Events
▼ 
const EventEmitter = require('events');

// Create custom event emitter
class MyEmitter extends EventEmitter {}

const myEmitter = new MyEmitter();

// Multiple listeners for same event
myEmitter.on('data', (data) => {
    console.log('Listener 1:', data);
});

myEmitter.on('data', (data) => {
    console.log('Listener 2:', data);
});

// One-time listener
myEmitter.once('connection', () => {
    console.log('Connected! (only fires once)');
});

// Emit events
myEmitter.emit('data', { message: 'Hello' });
// Output: Listener 1: { message: 'Hello' }
//         Listener 2: { message: 'Hello' }

myEmitter.emit('connection');
myEmitter.emit('connection'); // Won't trigger listener again

// Error handling - CRITICAL!
myEmitter.on('error', (err) => {
    console.error('Error occurred:', err.message);
});

myEmitter.emit('error', new Error('Something went wrong'));

// Remove listeners to prevent memory leaks
const listener = (data) => console.log('Temp listener:', data);
myEmitter.on('temp', listener);
myEmitter.emit('temp', 'test');

// Clean up
myEmitter.removeListener('temp', listener);
// or: myEmitter.off('temp', listener);

// Check listener count
console.log('Listener count:', myEmitter.listenerCount('data')); // 2

// Increase max listeners if needed (default 10)
myEmitter.setMaxListeners(20);

// Async event handlers with captureRejections
const emitter = new EventEmitter({ captureRejections: true });

emitter.on('async-task', async (data) => {
    throw new Error('Async error');
});

emitter.on('error', (err) => {
    console.error('Caught async error:', err.message);

emitter.emit('async-task', { id: 1 });

const EventEmitter = require('events');

// Banking Transaction Event Emitter
class BankingService extends EventEmitter {
    constructor() {
        super();
        this.accounts = new Map([
            ['ACC001', { balance: 50000, holder: 'John Doe' }],
            ['ACC002', { balance: 30000, holder: 'Jane Smith' }]
        ]);
        this.transactionLog = [];
    }

    async deposit(accountId, amount) {
        try {
            const account = this.accounts.get(accountId);
            if (!account) {
                throw new Error(`Account ${accountId} not found`);
            }

            account.balance += amount;
            
            const transaction = {
                id: `TXN${Date.now()}`,
                type: 'DEPOSIT',
                accountId,
                amount,
                newBalance: account.balance,
                timestamp: new Date()
            };

            this.transactionLog.push(transaction);
            
            // Emit transaction event
            this.emit('transaction', transaction);
            this.emit('deposit', transaction);
            
            return transaction;
        } catch (error) {
            this.emit('error', error);
            throw error;
        }
    }

    async withdraw(accountId, amount) {
        try {
            const account = this.accounts.get(accountId);
            if (!account) {
                throw new Error(`Account ${accountId} not found`);
            }

            if (account.balance < amount) {
                const error = new Error('Insufficient funds');
                this.emit('insufficient-funds', { accountId, amount, balance: account.balance });
                throw error;
            }

            account.balance -= amount;
            
            const transaction = {
                id: `TXN${Date.now()}`,
                type: 'WITHDRAWAL',
                accountId,
                amount,
                newBalance: account.balance,
                timestamp: new Date()
            };

            this.transactionLog.push(transaction);
            
            this.emit('transaction', transaction);
            this.emit('withdrawal', transaction);
            
            // Emit alert for large withdrawals
            if (amount > 10000) {
                this.emit('large-transaction', transaction);
            }
            
            return transaction;
        } catch (error) {
            this.emit('error', error);
            throw error;
        }
    }
}

const bankingService = new BankingService();

    }

        console.log(`[ALERT] Insufficient funds for ${data.accountId}. Requested: $${data.amount}, Available: $${data.balance}`);
    }
}

🎯 Key Takeaways

  • Observer Pattern: EventEmitter implements publish-subscribe for decoupled communication
  • Synchronous Execution: Listeners called synchronously in registration order (use setImmediate for async)
  • Error Events: Always listen for 'error' - unhandled errors crash the process
  • Memory Leaks: Default max 10 listeners per event. Remove listeners when done to prevent leaks
  • once() Method: Use for one-time events (auto-removes after first emission)
  • Multiple Listeners: Same event can have multiple listeners (all executed)
  • captureRejections: Enable to catch async errors in event handlers automatically
  • Use Cases: Streams, HTTP servers, custom app events, decoupling services

✅ DO's

  • Always add 'error' event listener
  • Remove listeners when no longer needed
  • Use once() for one-time events
  • Name events consistently (kebab-case)
  • Document emitted events in class
  • Use setMaxListeners if legitimate need
  • Emit events with descriptive data objects
  • Keep listeners focused and small

❌ DON'Ts

  • Don't forget error event handlers
  • Don't add infinite listeners (memory leak)
  • Don't modify emitter in listener callbacks
  • Don't use events for synchronous flow control
  • Don't catch errors inside emit() calls
  • Don't rely on listener execution order critically
  • Don't use events for return values
  • Don't ignore max listeners warnings
Q44 Event-Driven Architecture - Message Queues (RabbitMQ, Kafka) Advanced
▼

🐰 RabbitMQ

Message broker using AMQP protocol. Exchanges route messages to queues based on routing keys. Types: direct, topic, fanout, headers. Supports acknowledgments, dead-letter queues, message TTL. Library: amqplib.

🌊 Apache Kafka

Distributed streaming platform with high throughput. Topics divided into partitions for parallel processing. Consumer groups enable load balancing. Retains messages for configured time (log-based). Library: kafkajs.

📨 Queue Patterns

Work Queue: multiple consumers compete for tasks. Pub/Sub: broadcast to multiple subscribers. Request/Reply: RPC pattern with correlation IDs. Priority queues: process high-priority messages first.

✅ Message Acknowledgments

Manual ack ensures message processed before removal. Auto-ack for performance (risk of loss). Negative ack (nack) to reject and requeue. Idempotent consumers handle duplicates. At-least-once vs exactly-once delivery.

💻 RabbitMQ Producer & Consumer with Work Queue
▼ 
// npm install amqplib
const amqp = require('amqplib');

// Producer - Send messages to queue
async function sendToQueue(queueName, message) {
    try {
        const connection = await amqp.connect('amqp://localhost');
        const channel = await connection.createChannel();
        
        // Declare queue (durable = survives broker restart)
        await channel.assertQueue(queueName, { durable: true });
        
        // Send message (persistent = survives broker restart)
        channel.sendToQueue(
            queueName, 
            Buffer.from(JSON.stringify(message)),
            { persistent: true }
        );
        
        console.log(`Sent: ${JSON.stringify(message)}`);
        
        await channel.close();
        await connection.close();
    } catch (error) {
        console.error('Send error:', error);
    }
}

// Consumer - Process messages from queue
async function consumeFromQueue(queueName, processMessage) {
    try {
        const connection = await amqp.connect('amqp://localhost');
        const channel = await connection.createChannel();
        
        await channel.assertQueue(queueName, { durable: true });
        
        // Prefetch: only 1 message at a time (fair dispatch)
        channel.prefetch(1);
        
        console.log(`Waiting for messages in ${queueName}...`);
        
        channel.consume(queueName, async (msg) => {
            if (msg) {
                const content = JSON.parse(msg.content.toString());
                console.log(`Received: ${JSON.stringify(content)}`);
                
                try {
                    await processMessage(content);
                    
                    // Acknowledge message (remove from queue)
                    channel.ack(msg);
                    console.log('Message processed and acknowledged');
                } catch (error) {
                    console.error('Processing error:', error);
                    
                    // Negative ack - requeue message
                    channel.nack(msg, false, true);
                }
            }
        }, { noAck: false }); // Manual acknowledgment
        
    } catch (error) {
        console.error('Consumer error:', error);
    }
}

// Pub/Sub with Topic Exchange
async function publishToExchange(exchangeName, routingKey, message) {
    const connection = await amqp.connect('amqp://localhost');
    const channel = await connection.createChannel();
    
    // Declare topic exchange
    await channel.assertExchange(exchangeName, 'topic', { durable: true });
    
    channel.publish(
        exchangeName,
        routingKey,
        Buffer.from(JSON.stringify(message)),
        { persistent: true }
    );
    
    console.log(`Published to ${exchangeName} with key ${routingKey}`);
    
    await channel.close();
    await connection.close();
}

async function subscribeToTopic(exchangeName, bindingKeys) {
    const connection = await amqp.connect('amqp://localhost');
    const channel = await connection.createChannel();
    
    await channel.assertExchange(exchangeName, 'topic', { durable: true });
    
    // Create exclusive queue (deleted when connection closes)
    const q = await channel.assertQueue('', { exclusive: true });
    
    // Bind queue to exchange with patterns
    for (const key of bindingKeys) {
        await channel.bindQueue(q.queue, exchangeName, key);
    }
    
    console.log(`Subscribed to ${bindingKeys.join(', ')}`);
    
    channel.consume(q.queue, (msg) => {
        console.log(`[${msg.fields.routingKey}] ${msg.content.toString()}`);
    }, { noAck: true });
}

🎯 Key Takeaways

  • RabbitMQ: Message broker with exchanges (direct, topic, fanout) and queues. Good for task distribution
  • Kafka: Distributed log with high throughput. Topics have partitions. Good for event streaming
  • Acknowledgments: Manual ack ensures at-least-once delivery. Auto-ack for performance but risk loss

  • Ordering: Kafka guarantees order within partition. RabbitMQ guarantees order per queue
  • Idempotency: Design consumers to handle duplicate messages safely
  • Dead Letter: Route failed messages to dead-letter queue for analysis

✅ DO's

  • Use manual acknowledgments for reliability
  • Implement idempotent consumers
  • Set message TTL to prevent queue buildup
  • Use dead-letter queues for failed messages

  • Use partitioning keys for ordering (Kafka)
  • Implement retry logic with exponential backoff
  • Enable message persistence for durability

❌ DON'Ts

  • Don't use auto-ack without understanding risks
  • Don't process messages without error handling
  • Don't ignore message ordering requirements
  • Don't forget to set consumer prefetch limits
  • Don't store large payloads in messages
  • Don't ignore consumer lag monitoring
  • Don't requeue failed messages infinitely
  • Don't share connections between threads unsafely
Q45 Serverless Architecture - AWS Lambda Functions Advanced

⚡ Lambda Basics

Event-driven serverless compute. Handler function receives event and context objects. Supports Node.js, Python, Java, etc. Auto-scaling, pay-per-invocation. Max timeout: 15 minutes. Memory: 128MB-10GB.

🥶 Cold Starts

First invocation or after idle period requires initialization (container creation, code loading). Warm starts reuse existing containers. Reduce cold starts: Provisioned Concurrency, smaller packages, optimize init code, use compiled languages.

📤 Event Sources

Trigger types: API Gateway (HTTP), S3 (file uploads), DynamoDB Streams, SQS/SNS, EventBridge, CloudWatch Events. Sync (HTTP) vs Async (S3, SNS). Stream-based (DynamoDB, Kinesis) use batching.

📦 Lambda Layers

Share code/dependencies across functions. Max 5 layers per function, 50MB zipped per layer. Common uses: npm packages, shared utilities, runtime dependencies. Reduces deployment package size.

        if (!accountId || !type || !amount) {
            return response(400, { error: 'Missing required fields' });
        }
        
        if (amount <= 0) {
            return response(400, { error: 'Amount must be positive' });
        }
        
        // Get account
        const account = await getAccount(accountId);
        if (!account) {
            return response(404, { error: 'Account not found' });
        }
        
        // Check balance for withdrawals
        if (type === 'WITHDRAWAL' && account.balance < amount) {
            return response(400, { 
                error: 'Insufficient funds',
                balance: account.balance,
                requested: amount
            });
        }
        
        // Calculate new balance
        const newBalance = type === 'DEPOSIT' 
            ? account.balance + amount 
            : account.balance - amount;
        
        // Create transaction record
        const transaction = {
            transactionId: uuidv4(),
            accountId,
            type,
            amount,
            oldBalance: account.balance,
            newBalance,
            timestamp: new Date().toISOString(),
            status: 'COMPLETED'
        };
        
        // Update account and save transaction (in transaction)
        await processTransactionAtomic(accountId, newBalance, transaction);
        
        console.log(`Transaction ${transaction.transactionId} completed`);
        
        return response(201, {
            message: 'Transaction successful',
            transaction
        });
        
    } catch (error) {
        console.error('Error:', error);
        return response(500, { 
            error: 'Internal server error',
            message: error.message

const jwt = require('jsonwebtoken');

const SECRET_KEY = process.env.JWT_SECRET || 'your-secret-key';

exports.handler = async (event, context) => {
    console.log('Auth event:', JSON.stringify(event));
    
    try {
        // Extract token from Authorization header
        const token = event.authorizationToken || event.headers?.Authorization;
        
        if (!token) {
            throw new Error('No token provided');
        }
        
        // Remove "Bearer " prefix if present
        const cleanToken = token.replace('Bearer ', '');
        
        // Verify JWT token
        const decoded = jwt.verify(cleanToken, SECRET_KEY);
        
        console.log('Decoded token:', decoded);
        
        // Generate IAM policy
        const policy = generatePolicy(
            decoded.userId,
            'Allow',
            event.methodArn,
            {
                userId: decoded.userId,
                email: decoded.email,
                role: decoded.role
            }
        );
        
        return policy;
        
    } catch (error) {
        console.error('Authorization failed:', error.message);
        
        // Return Deny policy
        throw new Error('Unauthorized'); // Returns 401
    }
};

function generatePolicy(principalId, effect, resource, context = {}) {
    const authResponse = {
        principalId

            Statement: [

                    Action: 'execute-api:Invoke',
                    Effect: effect,
                    Resource: resource
                }
            ]
        };
    }
    
    // Pass user context to Lambda function
    authResponse.context = context;
    
    return authResponse;
}

function apiResponse(statusCode, body) {
    return {
        statusCode,
        headers: {
            'Content-Type': 'application/json',
            'X-Request-Id': context.requestId
        },
        body: JSON.stringify(body)
    };
}

🎯 Key Takeaways

  • Gateway Types: REST API (full features), HTTP API (70% cheaper, lower latency), WebSocket API
  • Throttling: Burst limit (concurrent) + Rate limit (per second). Prevents backend overload
  • Authorizers: Lambda (custom), Cognito (JWT), IAM (AWS credentials). Cache for performance
  • Caching: 0.5GB-237GB cache. TTL 0-3600s. Cache key from path/query/headers
  • Stages: Dev, staging, prod with different configs. Deploy to stages independently
  • Request Validation: JSON Schema validation before Lambda invocation. Reduces costs
  • CORS: Configure for browser clients. Enable in API Gateway or Lambda responses
  • Pricing: Per million requests. HTTP API ~70% cheaper than REST API

✅ DO's

  • Enable caching for read-heavy endpoints
  • Use request validation to reduce Lambda costs
  • Implement proper throttling limits per stage
  • Cache authorizer responses (300s recommended)
  • Use HTTP API for simple REST APIs (cheaper)
  • Enable CloudWatch logging for debugging
  • Configure CORS properly for browser clients
  • Use custom domain names with SSL certificates

❌ DON'Ts

  • Don't skip throttling configuration (prevents abuse)
  • Don't cache responses with sensitive data
  • Don't use API Keys as sole authentication
  • Don't forget to enable CORS for browser APIs
  • Don't ignore 429 (rate limit) errors
  • Don't skip request validation (reduces costs)
  • Don't use REST API when HTTP API suffices
  • Don't expose internal error details to clients
Q47 WebSockets - Real-time Bidirectional Communication Intermediate
▼

🔌 Socket.io Basics

Real-time bidirectional communication over WebSocket protocol (fallback to HTTP long-polling). Server and client emit/listen for events. Persistent connection reduces latency vs HTTP polling. Library: socket.io (server + client).

📡 Events & Rooms

Custom events with data payloads. Rooms group sockets for targeted messaging (chat rooms, game lobbies). Broadcasting to all/room/specific sockets. Namespaces for logical separation (/admin, /user).

🏠 Room Management

Clients join/leave rooms dynamically. Emit to room: io.to('room').emit(). Private rooms for 1-on-1 chat. Room data stored in memory (use Redis adapter for multi-server).

📢 Broadcasting Patterns

broadcast.emit() - all except sender. io.emit() - all clients. socket.emit() - single client. to('room').emit() - room members. Acknowledgment callbacks for confirmation.

const express = require('express');
const { createServer } = require('http');
const { Server } = require('socket.io');

const app = express();
const httpServer = createServer(app);
const io = new Server(httpServer, {
    cors: {
        origin: '*',
        methods: ['GET', 'POST']
    }
});

// Track active users
const activeUsers = new Map();

// Connection event
io.on('connection', (socket) => {
    console.log(`Client connected: ${socket.id}`);
    
    // User joins with username
    socket.on('join', (username) => {
        activeUsers.set(socket.id, username);
        
        // Broadcast to all other clients
        socket.broadcast.emit('user-joined', {
            username,
            userId: socket.id,
            timestamp: new Date()
        });
        
        // Send current user list to new user
        socket.emit('active-users', Array.from(activeUsers.values()));
        
        console.log(`${username} joined. Total users: ${activeUsers.size}`);
    });
    
    // Join a room
    socket.on('join-room', (roomId) => {
        socket.join(roomId);
        console.log(`${socket.id} joined room: ${roomId}`);
        
        // Notify room members
        io.to(roomId).emit('room-notification', {
            message: `${activeUsers.get(socket.id)} joined the room`,
            roomId
        });
    });
    
    // Leave a room
    socket.on('leave-room', (roomId) => {
        socket.leave(roomId);
        console.log(`${socket.id} left room: ${roomId}`);
        
        io.to(roomId).emit('room-notification', {
            message: `${activeUsers.get(socket.id)} left the room`,
            roomId
        });
    });
    
    // Send message to room
    socket.on('room-message', ({ roomId, message }) => {
        const username = activeUsers.get(socket.id);
        
        // Emit to everyone in room including sender
        io.to(roomId).emit('room-message', {
            username,
            userId: socket.id,
            message,
            roomId,
            timestamp: new Date()
        });
    });
    
    // Private message (1-on-1)
    socket.on('private-message', ({ toUserId, message }) => {
        const fromUsername = activeUsers.get(socket.id);
        
        // Send to specific socket
        io.to(toUserId).emit('private-message', {
            fromUserId: socket.id,
            fromUsername,
            message,
            timestamp: new Date()
        });
    });
    
    // Event with acknowledgment callback
    socket.on('send-data', (data, callback) => {
        console.log('Received data:', data);
        
        // Acknowledge receipt
        callback({
            status: 'received',
            timestamp: new Date()
        });
    });
    
    // Disconnect event
    socket.on('disconnect', (reason) => {
        const username = activeUsers.get(socket.id);
        activeUsers.delete(socket.id);
        
        // Notify all users
        io.emit('user-left', {
            username,
            userId: socket.id,
            reason,
            timestamp: new Date()
        });
        
        console.log(`${username} disconnected. Reason: ${reason}`);
    });
});

httpServer.listen(3000, () => {
    console.log('Socket.io server running on port 3000');
});

const io = require('socket.io')(3000);

// Middleware for authentication
io.use(async (socket, next) => {
    const token = socket.handshake.auth.token;
    if (!token) return next(new Error('Authentication required'));
    
    try {
        const user = await verifyToken(token);
        socket.userId = user.id;
        socket.accountIds = user.accounts;
        next();
    } catch (err) {
        next(new Error('Invalid token'));
    }
});

io.on('connection', (socket) => {
    console.log(`User ${socket.userId} connected`);
    
    // Subscribe to account updates
    socket.accountIds.forEach(accountId => {
        socket.join(`account:${accountId}`);
    });
    
    // Get real-time balance
    socket.on('subscribe-balance', (accountId) => {
        if (!socket.accountIds.includes(accountId)) {
            return socket.emit('error', { message: 'Unauthorized' });
        }
        socket.join(`account:${accountId}`);
        // Send current balance immediately
        sendCurrentBalance(socket, accountId);
    });
    
    socket.on('disconnect', () => {
        console.log(`User ${socket.userId} disconnected`);
    });
});

// Emit balance update when transaction occurs
function notifyBalanceUpdate(accountId, newBalance, transaction) {
    io.to(`account:${accountId}`).emit('balance-updated', {
        accountId,
        balance: newBalance,
        transaction: {
            id: transaction.id,
            type: transaction.type,
            amount: transaction.amount,
            timestamp: transaction.timestamp
        }
    });
}

Q48 WebSockets Scaling - Redis Pub/Sub & Horizontal Scaling Advanced
▼

📡 Redis Adapter

Broadcast events across multiple Socket.io servers using Redis pub/sub. All server instances subscribe to Redis channels. When one server emits to a room, Redis propagates to all servers. Library: @socket.io/redis-adapter.

🔗 Sticky Sessions

Load balancer ensures client always connects to same server instance. Required because WebSocket = stateful connection. Use IP hash or cookie-based routing. NGINX ip_hash or AWS ALB target group stickiness.

⚖️ Load Balancing

Distribute WebSocket connections across servers. ALB supports WebSocket. NGINX with proxy_pass and upgrade headers. HAProxy with balance source for sticky sessions. Monitor connection distribution.

🚀 Horizontal Scaling

Add more server instances to handle load. Redis stores shared state (rooms, sessions). Use Redis Cluster for high availability. Auto-scaling based on connection count or CPU. Graceful shutdown to drain connections.

🎯 Key Takeaways

  • Redis Adapter: Enables broadcasting across multiple Socket.io instances via pub/sub
  • Sticky Sessions: Client must reconnect to same server. Use IP hash or cookies
  • Load Balancer: Must support WebSocket upgrade (NGINX, ALB, HAProxy)
  • Shared State: Store rooms/sessions in Redis, not server memory
  • Graceful Shutdown: Drain connections before killing server instance
  • Monitoring: Track connection count per server, Redis latency, message throughput

✅ DO's

  • Use Redis adapter for multi-server setups
  • Enable sticky sessions on load balancer
  • Implement graceful shutdown for connections
  • Monitor connection distribution across servers

❌ DON'Ts

  • Don't scale without Redis adapter (events won't broadcast)
  • Don't forget sticky sessions (clients disconnect)
  • Don't store state in server memory only
  • Don't ignore Redis single point of failure
Q49 Docker Containerization - Building & Deploying Node.js Apps Intermediate
▼

📦 Dockerfile

Blueprint for Docker image. FROM (base image), COPY (files), RUN (commands), CMD (startup). Layer caching for faster builds. Use .dockerignore to exclude node_modules, .git. Optimize layer order.

🏗️ Multi-stage Builds

Multiple FROM statements for build and runtime. Build stage installs devDependencies, compiles TypeScript. Production stage copies only needed files. Reduces final image size significantly (100MB+ savings).

💾 Volumes & Bind Mounts

Persist data outside container lifecycle. Named volumes for databases, logs. Bind mounts for development (live reload). Volume drivers for cloud storage (AWS EFS, Azure Files).

🌐 Networks

Container-to-container communication. Bridge network (default), Host network, Overlay network (Swarm). Service discovery via container names. Port mapping (-p 3000:3000) for external access.

🎯 Key Takeaways

  • Multi-stage: Build in one stage, run in another. Smaller production images
  • Layer Caching: Order Dockerfile commands to maximize cache hits (COPY package.json first)
  • Security: Run as non-root user. Scan images for vulnerabilities. Use official base images
  • Volumes: Persist data (databases, logs, uploads). Survive container restarts
  • Environment: Use ENV or .env files for configuration. Never hardcode secrets
  • .dockerignore: Exclude node_modules, .git, tests to speed up builds

✅ DO's

  • Use multi-stage builds for production
  • Run as non-root user (USER node)
  • Use .dockerignore to exclude unnecessary files
  • Tag images with version numbers

❌ DON'Ts

  • Don't install devDependencies in production
  • Don't run as root user (security risk)
  • Don't hardcode secrets in Dockerfile
  • Don't use 'latest' tag in production
Q50 Docker Compose - Multi-Container Applications Intermediate

📄 docker-compose.yml

YAML file defining multi-container app. services (containers), networks, volumes. Version 3.x syntax. Define dependencies with depends_on. Environment variables, port mappings, restart policies.

depends_on ensures start order (DB before API). healthcheck verifies service ready. Wait scripts for database readiness. Links (deprecated) vs networks for communication.

Isolated network for services. Default bridge network auto-created. Custom networks for segmentation. Services communicate via service names (api → db:5432). External networks connect to existing networks.

💾 Volumes

Named volumes for data persistence. Share volumes between services. Volume drivers for NFS, cloud storage. Backup strategies. Development bind mounts for hot reload.

🎯 Key Takeaways

  • depends_on: Control service startup order. Add healthcheck for readiness
  • Networks: Services communicate via service names. Isolated from host
  • Volumes: Persist database data, logs. Survive container recreation
  • Commands: up (start), down (stop & remove), logs, ps, exec
  • Environment: Use .env file for secrets. env_file or environment keys

✅ DO's

  • Define health checks for critical services
  • Use named volumes for data persistence
  • Separate dev and prod compose files

❌ DON'Ts

  • Don't rely on depends_on for readiness
  • Don't use host network mode unnecessarily
  • Don't forget to backup volumes
