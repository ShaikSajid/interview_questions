Q51 CI/CD - GitHub Actions for Node.js Advanced
▼

🔄 Workflow Fundamentals

GitHub Actions automates CI/CD pipelines with YAML workflows. Triggers: push, PR, schedule, manual. Jobs run in parallel, steps sequential. Runners: GitHub-hosted or self-hosted. Secrets for credentials.

🧪 Automated Testing

Run unit, integration, E2E tests on every commit. Setup services (PostgreSQL, Redis) in workflow. Coverage thresholds enforce quality. Parallel matrix testing across Node versions. Cache dependencies for speed.

🐳 Docker Build & Push

Build Docker images in pipeline. Multi-stage builds for optimization. Push to GHCR, ECR, or DockerHub. Scan images with Trivy for vulnerabilities. Tag with branch, SHA, semantic version.

🚀 Multi-Environment Deployment

Deploy to dev/staging/prod environments. Environment protection rules for prod. AWS ECS/Lambda deployment. Smoke tests after deployment. Rollback on failure. Slack/email notifications.

🎯 Key Takeaways

  • Workflows: YAML files in .github/workflows/, triggered by repo events
  • Jobs: Run in parallel by default. Use needs: [job] for dependencies
  • Caching: Cache node_modules with actions/setup-node cache: 'npm'
  • Secrets: Store in repo settings, access via ${{ secrets.NAME }}
  • Services: Run databases/Redis as service containers for testing
  • Matrix: Test across multiple Node versions, OS platforms in parallel

✅ DO's

  • Cache dependencies for faster builds
  • Use environment protection for production
  • Run security scanning (Snyk, CodeQL)
  • Implement rollback mechanisms

❌ DON'Ts

  • Don't hardcode secrets in workflows
  • Don't skip tests for faster deployment
  • Don't deploy without health checks
  • Don't ignore failed builds
Q52 CI/CD - Blue-Green Deployment Advanced
▼

🔵🟢 Blue-Green Pattern

Two identical environments (blue=current, green=new). Deploy to green, test, switch traffic instantly. Zero downtime. Instant rollback if issues. Both environments share database. Higher cost (2x resources).

🔄 Traffic Switching

Load balancer switches traffic between environments. AWS ALB target groups, Kubernetes service selectors. Test green in production before switch. Monitor metrics after switch. Keep blue warm for rollback.

💾 Database Strategy

Backward-compatible migrations. Add nullable columns, never drop immediately. Multi-phase migrations. Feature flags hide new features. Shared database between blue/green. Test rollback scenarios.

📊 Health Checks

/health endpoint for liveness. /ready for readiness (DB checks). ALB health checks route traffic only to healthy targets. Graceful shutdown on SIGTERM. Drain connections before termination.

🎯 Key Takeaways

  • Zero Downtime: Traffic switches instantly, no service interruption
  • Instant Rollback: Switch back to blue in seconds if issues detected
  • Testing: Test green in production environment before switching traffic
  • Cost: Requires 2x resources during deployment (blue + green running)
  • Migrations: Must be backward-compatible for both versions
  • Monitoring: Watch error rates, latency for 5-10 min after switch

✅ DO's

  • Run comprehensive health checks on green
  • Keep blue running during monitoring period
  • Use feature flags for gradual rollouts
  • Automate rollback based on metrics

❌ DON'Ts

  • Don't make breaking database changes
  • Don't scale down blue immediately after switch
  • Don't skip smoke tests on green
  • Don't forget graceful shutdown handling
Q53 TypeScript - Basics with Node.js Intermediate
▼

📘 Type System

Static typing catches errors at compile-time. Basic types: string, number, boolean, array. Interfaces define object shapes. Enums for constants. Type aliases for complex types. Avoid 'any' type.

🔧 tsconfig.json

Configuration file for TypeScript compiler. strict: true enables all checks. target: ES2020, module: commonjs. outDir for compiled JS. sourceMap for debugging. noUnusedLocals, noImplicitReturns for quality.

🏗️ Classes & Interfaces

Classes with access modifiers (public, private, protected). Interfaces for contracts. Implements keyword for interface implementation. Abstract classes. Readonly properties. Constructor parameter properties.

🎯 Type Guards

Runtime type checking with type guards. typeof for primitives. instanceof for classes. Custom type guards with 'is' keyword. Union types narrowed by guards. Unknown type safer than any.

🎯 Key Takeaways

  • Type Safety: Catch errors at compile-time, not runtime
  • Interfaces: Define object shapes, can be extended and merged
  • Strict Mode: Enable all strict checks for maximum type safety
  • Generics: Write reusable code that works with multiple types
  • Type Guards: Narrow union types with runtime checks
  • Development: Better IDE support, autocomplete, refactoring

✅ DO's

  • Enable strict mode in tsconfig.json
  • Type all function parameters and returns
  • Use interfaces for object shapes
  • Prefer unknown over any type

❌ DON'Ts

  • Don't use 'any' type (defeats purpose)
  • Don't ignore compiler errors
  • Don't skip type annotations
  • Don't disable strict checks
Q54 TypeScript - Advanced Types Advanced
▼

🔀 Union & Intersection

Union types (A | B): value can be either type. Intersection (A & B): combine multiple types. Discriminated unions with literal types. Type narrowing in conditionals. Exhaustiveness checking with never.

🎭 Utility Types

Partial makes all properties optional. Required makes all required. Pick selects properties. Omit excludes properties. Record for key-value maps. ReturnType extracts return type.

🔑 Mapped Types

Transform properties of existing types. Readonly, Partial are built-in mapped types. Create custom mapped types with [K in keyof T]. Conditional types with extends. Template literal types for string manipulation.

🧬 Generics Advanced

Generic constraints with extends. Multiple type parameters. Default generic types. Generic classes and interfaces. Conditional types. infer keyword for type extraction. Variance in generics.

🎯 Key Takeaways

  • Union Types: Value can be one of several types (string | number)
  • Utility Types: Built-in helpers like Partial, Pick, Omit, Record
  • Mapped Types: Transform existing types by mapping over properties
  • Conditional Types: T extends U ? X : Y for type-level logic
  • Template Literals: String manipulation at type level
  • Type Inference: Let TypeScript infer types when possible

✅ DO's

  • Use utility types instead of rewriting logic
  • Leverage type inference when clear
  • Use discriminated unions for variants
  • Constrain generics appropriately

❌ DON'Ts

  • Don't over-complicate with complex types
  • Don't ignore type errors with any
  • Don't nest conditionals too deeply
  • Don't create unreadable type aliases
Q55 Monitoring - APM Tools Advanced
▼

📊 Application Performance Monitoring

APM tools track app performance in production. Distributed tracing across microservices. Request-level metrics (latency, throughput). Error tracking with stack traces. Popular: New Relic, DataDog, Dynatrace, AppDynamics.

🔍 Distributed Tracing

Track requests across multiple services. OpenTelemetry standard for instrumentation. Trace IDs propagate through headers. Spans represent operations. Flame graphs visualize latency. Identify bottlenecks.

📈 Metrics & Dashboards

RED metrics: Rate, Errors, Duration. Apdex score for user satisfaction. Custom business metrics. Real-time dashboards. Historical trends. Percentiles (p50, p95, p99) for latency. Memory, CPU usage.

🚨 Error Tracking

Capture uncaught exceptions. Stack traces with source maps. Error grouping by similarity. Impact analysis (affected users). Sentry, Rollbar for error tracking. Breadcrumbs for debugging context.

🎯 Key Takeaways

  • APM: End-to-end visibility into application performance
  • Tracing: Follow requests through microservices with trace IDs
  • Metrics: Track rate, errors, duration (RED), response times
  • Alerting: Proactive notifications on threshold breaches
  • Tools: DataDog, New Relic, OpenTelemetry, Prometheus
  • Overhead: APM adds 1-5% overhead, acceptable for value gained

✅ DO's

  • Instrument critical paths early
  • Set up dashboards for key metrics
  • Use distributed tracing for microservices
  • Track business metrics alongside technical

❌ DON'Ts

  • Don't wait for issues to add monitoring
  • Don't ignore APM overhead in performance
  • Don't track too many custom metrics
  • Don't forget to sanitize sensitive data
Q56 Monitoring - Alerting Strategies Advanced
▼

🔔 Alert Types

Threshold alerts (CPU > 80%). Anomaly detection (ML-based). Rate of change alerts (traffic spike). Composite alerts (multiple conditions). SLO-based alerts (error budget burn). Dead man's switch (heartbeat).

⚠️ Alert Severity

Critical: immediate action, page on-call. High: investigate within 30 min. Medium: check next business day. Low: informational. Escalation policies. Alert fatigue prevention. Group related alerts.

📞 Notification Channels

PagerDuty, OpsGenie for on-call. Slack for team notifications. Email for non-urgent. SMS for critical. Webhooks for custom integrations. Runbooks attached to alerts. Context in notifications.

🎯 Best Practices

Alert on symptoms, not causes. Actionable alerts only. Include relevant context (service, region). Define SLOs (99.9% uptime). Alert on SLO violations. Test alerts regularly. Review and tune thresholds.

🎯 Key Takeaways

  • Symptoms vs Causes: Alert on user-impacting issues, not internal metrics
  • Severity Levels: Critical (page), High (30min), Medium (next day)
  • Alert Fatigue: Too many alerts = ignored alerts. Keep signal high
  • SLO-Based: Alert on error budget burn rate, not arbitrary thresholds
  • Runbooks: Every alert should have action steps attached
  • Testing: Regularly test alert delivery and escalation paths

✅ DO's

  • Alert on user-impacting issues
  • Include context in alert messages
  • Define clear escalation policies
  • Review and tune alerts regularly

❌ DON'Ts

  • Don't alert on every metric spike
  • Don't create alerts without actions
  • Don't ignore alert fatigue
  • Don't forget to test alert delivery
Q57 Native Addons - C++ Integration Advanced
▼

⚙️ N-API Basics

Node-API (N-API) stable ABI for native modules. Write C/C++ code callable from Node.js. Use for CPU-intensive tasks, existing C++ libraries. node-gyp builds native modules. Works across Node versions.

🔧 Build Configuration

binding.gyp defines build config. Include directories, libraries. Cross-platform builds (Windows, Linux, Mac). node-addon-api C++ wrapper over N-API. Prebuilds for distribution. node-pre-gyp for binaries.

🔄 Data Conversion

Convert between JS and C++ types. Numbers, strings, objects, arrays. Buffer for binary data. Callbacks, promises from C++. Reference counting for memory management. Handle scope for GC.

⚡ Use Cases

Cryptography (fast hashing, encryption). Image processing (resize, filters). Financial calculations (high precision). Legacy C++ library integration. Hardware access. Performance-critical algorithms.

🎯 Key Takeaways

  • N-API: Stable ABI for native modules, works across Node versions
  • Performance: 10-100x faster for CPU-intensive operations
  • Use Cases: Crypto, image processing, math, legacy C++ integration
  • Build: node-gyp compiles C++ to .node binary addon
  • Memory: Manual memory management required in C++ code
  • Complexity: Harder to debug, platform-specific builds

✅ DO's

  • Use N-API for version compatibility
  • Profile before optimizing with C++
  • Handle errors gracefully
  • Provide prebuilt binaries

❌ DON'Ts

  • Don't use for I/O-bound tasks
  • Don't forget memory cleanup
  • Don't block event loop in C++
  • Don't ignore cross-platform issues
Q58 Native Addons - Performance Optimization Advanced
▼

🚀 Async Execution

Use AsyncWorker to avoid blocking event loop. libuv thread pool for background work. Return promises to JS. Progress callbacks for long operations. Cancel long-running tasks. ThreadSafeFunction for callbacks.

💾 Memory Optimization

Use Buffer for zero-copy data sharing. Avoid copying large objects. External ArrayBuffer for binary data. HandleScope for local references. Persistent handles for long-lived objects. Weak references for caches.

⚡ Algorithm Optimization

SIMD for parallel operations. OpenMP for multi-threading. Avoid frequent JS ↔ C++ crossings. Batch operations. Use native data structures. Profile with perf, valgrind. Compiler optimizations (-O3).

🔍 Profiling & Debugging

node --inspect for debugging. Chrome DevTools for C++ code. Valgrind for memory leaks. perf for CPU profiling. Address sanitizer for buffer overflows. Benchmark.js for performance tests.

🎯 Key Takeaways

  • AsyncWorker: Run C++ code off event loop with libuv threads
  • Zero-Copy: Use Buffer.from() to avoid data copying between JS/C++
  • Threading: OpenMP, std::thread for CPU parallelism in C++
  • Memory: Manual cleanup required, use RAII patterns
  • Profiling: Use perf, valgrind to find bottlenecks
  • Benchmarks: Always measure before/after optimization

✅ DO's

  • Use AsyncWorker for long operations
  • Leverage Buffer for binary data
  • Profile before optimizing
  • Test on all target platforms

❌ DON'Ts

  • Don't block event loop with sync C++ calls
  • Don't copy large data between JS/C++
  • Don't forget to test memory leaks
  • Don't optimize without profiling first
Q59 Production - Best Practices Advanced
▼

🔐 Security Hardening

Run as non-root user. Helmet.js for headers. Rate limiting with express-rate-limit. Input validation with Joi. Dependency scanning (npm audit, Snyk). Environment variable secrets. HTTPS only. JWT with short expiry.

📊 Observability

Structured logging (JSON format). Correlation IDs for requests. Metrics (Prometheus). Distributed tracing (OpenTelemetry). Health checks (/health, /ready). APM tools (DataDog). Log aggregation (ELK, CloudWatch).

⚡ Performance

Clustering for multi-core. PM2 for process management. Caching (Redis, in-memory). Database connection pooling. Compression (gzip). CDN for static assets. Optimize database queries. Index frequently queried fields.

🛡️ Reliability

Graceful shutdown (SIGTERM handling). Circuit breakers for external calls. Retry with exponential backoff. Timeouts on all requests. Health checks for dependencies. Auto-restart on crashes (PM2). Blue-green deployments.

🎯 Key Takeaways

  • Security: Helmet, rate limiting, input validation, dependency scanning
  • Logging: Structured JSON logs with correlation IDs, sanitize sensitive data
  • Clustering: Use all CPU cores with cluster module or PM2
  • Graceful Shutdown: Handle SIGTERM, close connections, drain requests
  • Health Checks: /health for liveness, /ready for readiness
  • Monitoring: APM, metrics, alerts on SLO violations

✅ DO's

  • Use PM2 or cluster for multiple instances
  • Implement graceful shutdown
  • Monitor error rates, latency
  • Use environment variables for config

❌ DON'Ts

  • Don't run as root user
  • Don't log sensitive data (passwords, tokens)
  • Don't skip dependency updates
  • Don't ignore memory leaks
Q60 Production - Scalability Strategies Advanced
▼

📈 Horizontal Scaling

Add more servers to handle load. Load balancer (ALB, NGINX) distributes traffic. Stateless services scale easily. Session in Redis, not memory. Auto-scaling based on CPU/memory. ECS, Kubernetes for orchestration.

⚡ Vertical Scaling

Increase server resources (CPU, RAM). Easier but has limits. Use clustering for multi-core. Less complex than horizontal. Cost increases linearly. Use for databases (RDS, Aurora).

🗄️ Database Scaling

Read replicas for read-heavy workloads. Sharding for write scaling. Connection pooling (pg.Pool). Database indexing for queries. Caching frequently accessed data. Materialized views. Partition large tables.

🔄 Caching Strategy

Redis for distributed cache. Cache-aside pattern. Write-through caching. TTL for expiration. Cache invalidation strategy. CDN for static assets. In-memory cache for hot data. Avoid over-caching.

🎯 Key Takeaways

  • Horizontal: Add more servers, easier to scale infinitely
  • Vertical: Bigger servers, simpler but limited scaling
  • Stateless: No session in server memory = easy horizontal scaling
  • Caching: Redis for distributed cache, reduces database load
  • Database: Read replicas, sharding, connection pooling
  • Load Balancing: Distribute traffic, health checks, sticky sessions

✅ DO's

  • Design stateless services for horizontal scaling
  • Use Redis for session management
  • Implement caching at multiple layers
  • Monitor and auto-scale based on metrics

❌ DON'Ts

  • Don't store session in server memory
  • Don't over-cache (memory limits)
  • Don't ignore database query optimization
  • Don't forget cache invalidation strategy
